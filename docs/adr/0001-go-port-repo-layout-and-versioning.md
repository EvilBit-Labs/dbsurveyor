# ADR 0001: Go Port Repository Layout and Versioning

- Status: Proposed (acceptance by maintainer closes issue #218)
- Date: 2026-07-09
- Related: #215 (spec), #218 (decision task), #216 (Phase 0 epic)

## Context

Issue #215 specifies reimplementing dbsurveyor in Go. Open question 12.1 asks
whether the Go implementation should live in a new `dbsurveyor-go` repository
or in a `go/` subtree of this repository during the transition. Open question
12.2 asks whether the cutover release should be versioned v0.2.0 or v1.0.0.

The migration strategy (see `docs/go-port-migration-strategy.md`) depends on
cheap interoperation between the two implementations throughout the port:

- Golden fixtures (#228) are generated by the Rust binaries and consumed by
  Go tests. They must not drift from the Rust source that produced them.
- The parity harness (#239) runs the Rust collector and the Go collector
  against the same database container and diffs the output.
- The encrypted-envelope byte specification (#224) is derived by reading the
  Rust source and doubles as documentation for the current implementation.
- All 37 tracking issues (#215-#251), the CodeRabbit configuration, and the
  project conventions live in this repository.

A separate repository would make each of these a cross-repo concern:
published fixture artifacts or vendored snapshots, cross-repo checkouts with
tokens in CI, and a split issue history mid-project. The only material cost
of the subtree is module-path aesthetics during the transition.

## Decision

1. Repository layout: the Go implementation is developed in a `go/` subtree
   of this repository. No new repository is created.
2. Module path: `github.com/EvilBit-Labs/dbsurveyor`, declared from the first
   commit even while the code lives under `go/`. Go permits the module root
   to differ from the repository root; all packages are under `internal/` and
   `cmd/`, so no external consumer depends on the interim path. This avoids a
   module rename at cutover.
3. Endgame: at cutover (Phase 6, #251) the Go tree is promoted to the
   repository root and the Rust tree is archived to a preserved branch
   (`rust-final`) with a README pointer. The repository identity, issue
   history, release history, and Homebrew Cask source remain continuous. No
   repository extraction is performed; `git subtree split` remains available
   later if a standalone repository is ever wanted.
4. Versioning: Rust releases are frozen at the v0.1.x line. The Go cutover
   release is v0.2.0 (not v1.0.0). Pre-releases along the way use
   v0.2.0-alpha.N / v0.2.0-rc.N tags. The v1.0.0 signal is reserved for after
   the Go implementation has survived real operator use. Because both
   implementations share one repository, there is no tag-collision concern.
5. CI isolation: workflows are paths-filtered so Rust CI ignores `go/**` and
   Go CI runs only on `go/**` (plus shared fixture paths), per #220.
6. Rust freeze enforcement: from the moment the Go scaffold lands (Phase 0),
   the Rust tree accepts security fixes only. Enforced mechanically: a CI
   check fails any pull request touching Rust source paths unless it carries
   the `rust-security-fix` label.

## Consequences

- Positive: single CI checkout for fixtures and the parity harness; one issue
  tracker and paper trail; continuous repository identity through cutover;
  no import-path churn; freeze policy is mechanical rather than remembered.
- Negative: the repository temporarily contains two toolchains, so CI
  configuration must stay strictly paths-filtered to keep runs fast and
  signals clean; contributors see both trees side by side during the
  transition (mitigated by README notes and the freeze check).
- Follow-up: #215 open questions 12.1 and 12.2 are struck once this ADR is
  accepted; #219 (scaffold) proceeds with the module path above.
