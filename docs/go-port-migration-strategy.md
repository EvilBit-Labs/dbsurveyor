# Go Port Migration Strategy

- Status: Proposed (companion to ADR 0001 and issue #215)
- Date: 2026-07-09
- Related: #215 (spec) and the phase epics #216, #217, #229, #230, #231,
  #232, #233

This document records the execution strategy for the Rust-to-Go
reimplementation: what gets built first, what runs in parallel, where the
hard gates are, and what is deliberately deferred. The issue tree under #215
tracks the work; this document explains the ordering rationale so the plan
survives context loss between sessions.

Summary: formats first, then a vertical slice, then breadth.

## 1. The contract: on-disk formats never change during the port

The single rule that converts a risky rewrite into a verifiable pipeline:
the JSON schema document, the zstd framing, and the AES-GCM `.enc` envelope
are frozen for the duration of the port. Every Go component is "done" when
it round-trips artifacts the Rust implementation produced - not when it
looks correct. Concretely:

- Golden fixtures generated by the Rust binaries are the arbiter (#228).
- The parity harness diffs Rust and Go collector output against the same
  database container (#239).
- No Go-only format additions ship until at least one release cycle after
  cutover (spec non-goal, #215 section 4).

## 2. Immediate actions (no Go code required)

Two tasks start before or alongside the scaffold because they need nothing
but the healthy Rust tree:

1. Generate the golden fixture corpus (#228 fixture-generation half): plain
   `.json` (single and multi-database), `.json.zst`, `.enc`, and combined
   compressed+encrypted `.enc`, with edge-case content (every `SampleStatus`
   variant, float quality scores, empty tables, non-ASCII data values,
   absent optional fields). Check the corpus into `go/testdata/golden/`
   with the generation script and Rust version recorded.
2. Write the encrypted-envelope byte specification (#224) from the Rust
   source in `dbsurveyor-core/src/security/`. This is the top-severity risk
   mitigation in #215 section 11 and is independently valuable documentation
   for the current implementation regardless of the port's fate.

## 3. Freeze Rust at Phase 0

As a single-maintainer project, the dominant risk is not technical - it is
maintaining two implementations while the port drags. From the moment the
Go scaffold lands:

- The Rust tree accepts security fixes only (mechanical enforcement per
  ADR 0001: CI fails Rust-touching PRs without the `rust-security-fix`
  label).
- Every mid-port feature idea becomes an issue tagged for post-cutover.
- Exactly two feature exceptions exist, because they are the reason for the
  port: the MSSQL adapter (#243) and the Oracle adapter (#244).

## 4. Vertical slice before breadth

The phase plan permits Phase 5 (postprocessor) to run any time after
Phase 1; this strategy makes it the deliberate path:

    Phase 0 -> Phase 1 -> Phase 2 (PostgreSQL) -> Phase 5 (postprocessor)
    -> tag v0.2.0-alpha.1 -> Phase 3 + Phase 4 (adapters, breadth)
    -> Phase 6 (cutover)

After Phase 2 plus Phase 5, the Go toolchain is complete for PostgreSQL:
collect, redact, analyze, generate markdown. That vertical slice is tagged
as a pre-release (v0.2.0-alpha.1) and dogfooded. Rationale:

- Real operator feedback on the Go binaries arrives months before parity is
  complete.
- The full pipeline (CLI ergonomics, output contracts, GoReleaser
  artifacts, Homebrew flow) is validated while the surface area is small.
- Phases 3 and 4 then become pure breadth - stamping out adapters against
  a proven harness - rather than load-bearing steps before anything is
  usable.

Within phases, the parallelism is real and recorded in the issue
dependencies: #236/#237/#238 after #235; the three Phase 3 adapters are
mutually independent; the two Phase 4 adapters are mutually independent;
#245/#246/#247 are mutually independent.

## 5. Port behavior, not code

The Go implementation is idiomatic Go that passes the parity gates - not a
transliteration of the Rust module structure. Where Rust idioms do not map
(serde attributes, the immutable `with_*` builder, `Zeroizing<String>`),
match the observable behavior and let the golden tests arbitrate. The
places to be pedantic are the recorded gotchas, because they are behavior:

- MySQL INFORMATION_SCHEMA mixed signedness (GOTCHAS 4.4).
- Nullable row-count estimates default to 0 (GOTCHAS 4.5).
- SQLite PRAGMA single-quote vs DML double-quote contexts (GOTCHAS 5.1),
  1-based `ordinal_position` (5.2), quoted `rowid` ordering (5.3).
- Batch-collection error counting checks errors before consuming results
  (GOTCHAS 4.2).
- Critical schema fields fail loudly on scan errors; only optional fields
  may default (GOTCHAS 1.5).

Credential handling follows #215 section 6.2: secrets as `[]byte` only,
best-effort zeroing, no secrets through fmt/log/error paths, and honest
documentation that deterministic zeroization is not claimed in Go. The hard
guarantee remains: credentials never appear in output files, enforced by
the recursive credential scan on every load path.

## 6. Cut over on the gate, not on a date

Phase 6 entry is mechanical (#249): the full parity suite green on all
release platforms, plus a one-time cross-implementation round-trip in both
directions (Go reads Rust artifacts; Rust reads Go artifacts). Until that
gate is green there is no cutover, regardless of how finished the port
feels. When it is green: release v0.2.0 (#250), promote the Go tree to the
repository root, archive the Rust tree to `rust-final`, and rewrite
README/AGENTS.md/GOTCHAS.md with each GOTCHAS entry re-validated (#251).

## 7. Sequence at a glance

1. #218 ADR accepted (ADR 0001).
2. Phase 0 (#216): scaffold, CI, GoReleaser, justfile - #218 -> #219 ->
   #220/#221/#222.
3. Immediately alongside: fixture corpus + envelope byte spec (#228, #224).
4. Phase 1 (#217): models, security, output, validation; exit on the
   golden-fixture suite (#228).
5. Phase 2 (#229): PostgreSQL adapter; exit on the parity harness (#239).
6. Phase 5 (#232): postprocessor; then tag v0.2.0-alpha.1 (vertical slice).
7. Phases 3 and 4 (#230, #231) in parallel: MySQL, SQLite, MongoDB, MSSQL,
   Oracle.
8. Phase 6 (#233): parity gate (#249) -> v0.2.0 release (#250) -> archive
   and docs rewrite (#251).
